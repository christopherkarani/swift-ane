# Lessons

## 2026-03-12

- When shipping dense SwiftUI desktop sidebars with custom glass panels and shadows, verify the left rail at the app's default window width. A sidebar that looks balanced at wide widths can still render visually clipped if the container width and inner margins are too tight.
- When cancelling Swift stream-reading tasks for `FileHandle.bytes.lines`, treat `CancellationError` as expected shutdown and never surface it in user-visible logs as a runtime failure.
- When a user explicitly names a design skill, run that skill’s required workflow before editing UI. Persist the generated design-system output, keep only the recommendations that fit the product type, and do not blindly import landing-page patterns into a desktop developer tool.
- After repackaging a macOS app bundle with the same bundle identifier, do not trust `open` alone for verification. macOS may foreground an already-running stale process, so kill or force a new instance before judging whether UI changes shipped.
- With Swift Observation, mutating a field inside a struct stored in an observed array is not reliable for live UI refresh. Copy the element out, mutate it, and assign it back into the array so logs and progress repaint while the process is still running.
- Do not do exploratory `URL` ancestor walking on the main-thread startup path of a packaged macOS app. Use a bounded string-path parent search and lazy-load filesystem-heavy state after the first window appears.
- In a macOS `HSplitView` sidebar, glass cards need width budget for both content and visual treatment. If a panel uses large corner radii and shadows, give the rail explicit container insets and a wider min/ideal width rather than relying on per-card padding inside a narrow column.
- For benchmark tooling aimed at engineers, expose comparison scope as an explicit execution-target choice (`Espresso Only` vs `vs Core ML`) instead of a negative implementation toggle like `ANE only`. Technical users scan for benchmark matrices and baseline targets, not backend flags.
- When a SwiftPM app package gains a test target whose directory already follows the default convention, avoid specifying a redundant custom `path:` unless needed. On 2026-03-12, removing the explicit path from `EspressoBenchAppTests` resolved a manifest planning failure during app packaging.
- For engineer-facing benchmark dashboards, raw comparison tables are not enough. Promote the fastest-baseline comparison into headline cards with explicit labels like `3.70x faster`, signed median/P95 deltas, and named baseline targets; keep the full matrix as a secondary audit view.
- If a benchmark dashboard has a derived comparison layer, do not hide that layer when the comparison dataset is absent or still running. Show a persistent placeholder state (`pending`, `espresso only`, or `missing data`) so users understand why the comparison metrics are not populated yet.
- Do not let comparison-mode benchmark runs proceed with a missing baseline model path. Validate the `.mlpackage` before launch and fail early in the UI; otherwise the CLI may continue with ANE-only results and make the dashboard look broken.
- For long-running benchmark logs in SwiftUI, avoid one main-actor hop per output line and avoid rendering the full growing log body live. Parse on the background task, batch chunks before UI submission, and render only a recent tail while the process is active.
- For long-running decode benchmarks, do not keep appending unbounded live log text into a SwiftUI view model that feeds a monospaced `Text` surface. Bound the log buffer or virtualize it; otherwise the app will appear to freeze even if the benchmark process is still healthy.
- If a desktop benchmark UI validates a required baseline model before launch, it also needs a first-class acquisition path for that model. A red validation banner without an in-app way to fix it reads as broken comparison support, not as a helpful prerequisite check.
- On macOS, a segmented `Picker` will still render its label unless you hide it explicitly. If the sidebar is narrow, that label can collapse into a vertical gutter and make the whole control look broken, so pair `.pickerStyle(.segmented)` with `.labelsHidden()` for compact technical panels.
- For engineer-facing desktop tools, a log console should not permanently consume dashboard space. Use an on-demand bottom drawer with `hidden`, `peek`, and `expanded` states so live process output is accessible without dominating the primary metrics view.
- When adding hardware telemetry to a benchmark UI, distinguish clearly between `not captured` and `not supported`. Power and thermal panels should render an explicit empty state plus the capture path (`powermetrics`, wrapper script, or adjacent artifact), not just disappear.
- For benchmark dashboards with live logs and rich summary views, do not store high-frequency console text on the same observed record that drives the full dashboard tree. Keep streamed log text in a dedicated observable path so console updates do not trigger broad selected-run re-evaluation.
- Swift Charts is not a free pass for benchmark traces. Large decode latency CSVs must be sampled or windowed before rendering, and the UI should tell engineers when the chart is showing a display sample rather than every raw point.
- On macOS, “default-on” power capture is a privilege problem before it is a UI problem. If a desktop app wants unattended `powermetrics` capture, design around a pre-authorized path such as passwordless sudo for `/usr/bin/powermetrics`, and surface that requirement explicitly in-app instead of implying capture will always work.
- For privilege-gated telemetry, a generic “not captured” banner is not enough. Persist and display the exact per-run capture failure reason so the dashboard can say `sudo: a password is required` or a similar concrete blocker instead of hiding the real cause behind generic guidance.
- If a dashboard card is the place where users discover an actionable setup failure, put the recovery control on that same card. Do not bury the only fix path inside a separate settings panel once the failure is already visible in context.
- When a privileged setup action changes future capability but cannot retroactively fix the selected artifact, the dashboard must say that explicitly. After authorizing `powermetrics`, tell the user the current run was not captured and that future runs are now ready, instead of leaving the old failure text in place.
- When direct ANE compilation fails with `ANE compile failed: no error` and later succeeds under slightly different timing, treat it as a transient compiler-service failure first, not as proof that the SwiftUI launcher is broken. Put bounded retry/backoff at the runtime compile boundary so both CLI and app runs benefit.
- If a published benchmark claim depends on a bespoke script/probe workflow, give the dashboard a dedicated run mode and parser for that contract. Do not try to squeeze a nested `run-*.json` artifact family through the generic `summary.json` / `espresso-bench` schema or the UI will look like comparison data is missing when the harness actually succeeded.
- When a user asks for Linear/Vercel styling, do not reuse the colorful benchmark-dashboard palette. Shift to near-black surfaces, restrained borders, white/gray typography, and one meaningful accent, then let the speedup headline carry the drama.
- For single-machine ANE speed demos, the most credible visual is a measured replay, not simultaneous live inference. Replaying both lanes from real timings preserves the `4.7x` story without ANE contention turning the showcase into a misleading benchmark.
- For Linear/Vercel-style SwiftUI desktop reskins, do not stop at colors. Default AppKit `bordered` buttons and bright metric numerics will keep the old aesthetic alive, so move actions onto shared neutral button styles and keep the card value hierarchy mostly white with accents limited to rails, dots, and borders.
- When a benchmark result is backed by checked-in artifacts and a reproducible workflow, do not call it a “claim” in the product UI. Use “verified benchmark,” “published result,” or “repro” language so the copy reflects the actual level of evidence.
- For narrow SwiftUI desktop sidebars, default to a single-column control rail with adaptive rows rather than dense side-by-side grids and fixed one-line path/action layouts. Use `ViewThatFits` for path rows, numeric controls, and section actions before increasing copy or padding.
- When a SwiftUI split-view desktop app still feels cramped after a sidebar redesign, fix the width budget before adding more adaptive content. Oversized `idealWidth` values, large outer insets, and visible segmented-picker labels can starve the detail pane even if the sidebar itself technically fits.
- For streaming SwiftUI demo text, do not repaint the entire source script plus a revealed overlay on every timer tick. Use a bounded trailing window around the live edge, remove non-essential progress chrome, and keep the newest text visible with bottom padding so the front of generation stays readable under load.
- For benchmark demo lanes, dense information beats oversized cards. Use a compact stat strip with the live rate first, and if you animate tok/s, tie the motion deterministically to the playback clock so it looks active while staying bounded and pausable.
- For benchmark showcase motion, add only evidence-linked animation: frontier caret, cadence milestones, elapsed time, lead gap, and sparkline values should all derive from the same playback clock. If a motion element cannot be explained by measured timing, it should not be in the demo.
- For screen-recorded benchmark demos, make the text surface behave like a chat assistant, not a terminal pane: pin short content to the bottom, auto-follow the live edge without expensive animated scrolling on every tick, and keep the text box short enough that hero metrics remain in frame.
- If a benchmark replay has multiple lanes with different speeds, model lane completion explicitly. Do not rely on text reveal clamping alone; cap the lane’s synthetic token count, freeze its live-rate animation, hide its caret, and surface a visible done state while slower lanes continue.
- If the user wants a feature directory hidden from Git “for now,” check whether those files are already tracked before relying on `.gitignore`. Ignore rules only hide untracked files; tracked files must be removed from the index with `git rm --cached`, which will appear as staged deletions while the local files remain present.
