# Espresso Benchmark Suite v1 — Design Document

**Date:** 2026-03-05
**Status:** Approved
**Purpose:** Measure ANE Direct inference performance vs Core ML baseline to inform company pivot decision.

## Scope

Three benchmarks at current ModelConfig dimensions (dim=768, hidden=2048, seqLen=256, heads=12):

1. **ANE Direct** — Forward pass via Espresso runtime (MIL → ANEKernel → IOSurface I/O)
2. **Core ML Baseline** — Same transformer layer via MLModel.prediction, three compute unit configs
3. **Power & Thermal** — 60-second sustained inference, thermal state tracking

Scaling tests deferred to v2 (requires parameterizing ModelConfig).

## Architecture

### Approach: Integrated SPM Target

New `EspressoBench` executable target in `Package.swift`. Imports `Espresso`, `ANERuntime`, `ANETypes`, `CPUOps`. Uses the real production pipeline — no mocking.

Core ML model generated by a Python coremltools script, loaded at benchmark time.

### File Structure

```
Sources/EspressoBench/
  main.swift              — Entry point, CLI arg parsing, orchestration
  BenchmarkRunner.swift   — Generic measurement harness (warmup, measure, stats)
  ANEDirectBench.swift    — Benchmark 1: ANE forward pass
  CoreMLBench.swift       — Benchmark 2: Core ML baseline
  ThermalMonitor.swift    — Benchmark 3: Thermal state tracking
  ResultsFormatter.swift  — Output formatting (text report + CSV)
  FLOPCalculator.swift    — FLOP counting for utilization percentages

scripts/
  generate_coreml_model.py  — coremltools script to create equivalent transformer

benchmarks/
  results/                — Output directory for benchmark runs
  models/                 — Generated .mlpackage files
```

## Benchmark 1: ANE Direct

1. Create random `LayerWeights` with `Float.random(in: -0.1...0.1)`
2. Create random input `TensorBuffer` (dim x seqLen)
3. Compile `LayerKernelSet` (5 kernels, single compilation)
4. Build `LayerSurfaceHandles` for cached surface access
5. Warmup: 50 forward passes
6. Measure: 1000 forward passes, each timed with `ContinuousClock`
7. Capture `StepTimingBreakdown` to separate ANE vs I/O vs CPU time
8. `os_signpost` markers for Instruments verification

**FLOPs per forward pass (current config):** ~3.83 GFLOPs

## Benchmark 2: Core ML Baseline

**Python script** generates `.mlpackage` with identical architecture via coremltools MIL builder.

**Swift benchmark** loads model and runs with three `MLComputeUnits` configs:
- `.all` (Core ML chooses)
- `.cpuAndNeuralEngine` (force ANE)
- `.cpuAndGPU` (GPU baseline)

Same warmup/measure cycle. Model load time measured separately.

## Benchmark 3: Thermal & Power

- In-process: `ProcessInfo.processInfo.thermalState` sampled every second during 60-second sustained run
- External: shell script wrapper for `sudo powermetrics` (documented, not automated)
- Triggered via `--sustained` CLI flag

## Measurement Details

- **Clock:** `ContinuousClock` (monotonic, nanosecond precision)
- **Warmup:** 50 iterations (absorbs JIT, cache warming, first-compile overhead)
- **Measured:** 1000 iterations
- **Statistics:** mean, median, P50, P95, P99, stddev, min, max
- **Signposts:** `os_signpost` on every forward pass for Instruments correlation

## Output

1. Human-readable report to stdout (format per spec)
2. CSV files with raw latencies to `benchmarks/results/YYYY-MM-DD-HHMMSS/`
3. Summary text file in same directory

## CLI Interface

```bash
.build/release/EspressoBench                        # Full suite
.build/release/EspressoBench --ane-only             # Skip Core ML
.build/release/EspressoBench --sustained            # 60-second thermal test
.build/release/EspressoBench --warmup 100 --iterations 5000
.build/release/EspressoBench --output benchmarks/results/my-run
```

## Success Criteria

- 2x+ latency improvement over Core ML → strong pivot signal
- 3x+ → exceptional
- Lower power during sustained → "faster AND efficient" story
- Consistent P95/P99 → production-viable
- >15% ANE utilization → improvement over maderix 11.2% baseline
